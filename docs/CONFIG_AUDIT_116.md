# Config Reconciliation Audit (#116)

> Deliverable for issue #116. Generated by the multi-agent audit workflow. The WARN->hard-fail flip remains OUT of scope; this is the gate before it.

> **Reading guide:** the **Summary**, **Prioritized fix list**, and **Full reconciliation matrix** below are the ORIGINAL pre-implementation snapshot (the gaps as found). For what was actually implemented vs deferred, the **Resolution status and corrections** section immediately below is authoritative — most P1/P2/P3 schema and doc gaps listed later are now RESOLVED on `development`.

## Summary

Union of ~95 config keys reconciled across six surfaces. The dominant structural fact is post-#115: JSON-schema validation is ADVISORY (warn-not-throw); only validateLogic() can hard-fail. So every schema-only rule (patterns, enums, maxima, integer types, conditional-required, additionalProperties) is currently non-blocking, and several do not have validateLogic backstops. Highest-risk items before the eventual hard-fail flip: (1) notifications.ntfy.url is required UNCONDITIONALLY so a disabled stub wrongly fails; (2) per-server servers[*].sync.autoRetry.* is read by code but has NO schema entry, so tightening additionalProperties would reject a valid override; (3) notifications.webhooks.telegram is read by code and shipped in the example as [] but is not in the schema (only survives because webhooks is open); (4) healthCheck.dashboard.auth has no conditional-required so basic/token auth with blank creds is silently accepted (auth-bypass-shaped); (5) validateLogic vs schema drift on sync.baseRetryDelayMs (no upper bound in validateLogic), per-server sync.* never validated, and cron validated only by field-count. additionalProperties:false should be applied repo-wide (only notifications.telegram has it today) AFTER the schema gaps above are closed, keeping generic[].headers open. Major doc drift: CONFIG.md still describes validation as hard-failing; healthCheck section, notifications.branding, telegram.chatIds/pollInterval, and ntfy.icon are undocumented; logging.redact is documented but unreachable from config (loggerConfig never forwards it); the ${VAR} substitution example is non-functional. Unraid surface is largely consistent (PUID/PGID 99/100 vs 1001 divergence is intentional and documented); minor gaps are DATA_DIR not in the template and compose's read-only single-file config mount defeating first-run bootstrap seeding.

## Resolution status and corrections

The list below is the raw audit output. Two of its characterisations were corrected on implementation, and the work was split:

- **Auth is fail-closed, not an open bypass.** The summary/fix-list call `healthCheck.dashboard.auth` with blank credentials "auth-bypass-shaped". Re-checking `dashboardAuth()` (`src/services/healthCheck.js`): a blank-credential `basic`/`token` config rejects every request (the submitted credential is compared against `undefined`), so the real failure mode is a misconfiguration **lockout**, not an open dashboard. The conditional-required schema fix still applies (it catches the misconfig early); the severity framing was the only error.
- **`notifications.webhooks.telegram` was legitimised, not removed.** It is a real (legacy) send path consumed by `notificationService`, so it was **described in the schema** (`botToken`+`chatId` required) rather than deleted. The `additionalProperties` note that says to set `webhooks` `false` "after removing that example key" is superseded: the key stays; `additionalProperties:false` for `webhooks` is deferred to #121.
- **Split:** the advisory→hard-fail flip and `additionalProperties:false` application are now tracked in **#121**. This ticket (#116) is reconciliation only.
- **Implemented in #116 (schema/doc accuracy; schema still advisory):**
  - *P1 (schema accuracy):* ntfy.url conditional, per-server `sync.autoRetry`, `webhooks.telegram[*]`, dashboard `auth` conditional-required, `webhooks.generic[*].url` non-empty, widened cron pattern, declared root `$schema`.
  - *P2 (code + docs):* forward `logging.redact` + `fileFormat` to the logger (`src/lib/loggerConfig.js`); CONFIG.md validation semantics (advisory vs hard-fail), `healthCheck` section, real `redact` default list, supported E2EE example; NOTIFICATIONS.md `branding` / telegram `chatIds`+`pollInterval` / `ntfy.icon` / `TELEGRAM_*` env fallbacks; example illustrates per-server overrides.
  - *P3 (hardening):* non-empty host for `servers[*].url`; rotation `maxSize`/`interval` patterns; maxima on `maxFiles`/perf thresholds/`pollInterval`; `minLength` on meaningful strings and array items.
  - *README + Unraid (2nd-round):* README validation wording reconciled (advisory now, hard-fail in #121); `ACTUAL_SYNC_SERVER_DATA_DIR` added to `unraid/actual-sync.xml` (was read by code but not exposed); CONFIG.md `baseRetryDelayMs` range corrected to 1000-10000.
  - *3rd-round (verified against the real libraries):* cron pattern now accepts the valid 6-field (seconds) form that node-schedule runs; rotation `maxSize` accepts two-letter units (`10MB`/`10KB`/`1GB`) that rotating-file-stream accepts, and both size/interval now require a positive magnitude (reject `0M`). Doc consistency: ARCHITECTURE.md schema-validation wording made advisory; stale `#116` hard-fail references in `configLoader.js` and `UNRAID_CA_PUBLISHING.md` repointed to `#121`; LOGGING.md rotation unit wording corrected.
  - All covered by `src/__tests__/configSchema.test.js` and `src/__tests__/loggerConfig.test.js`.
- **Deferred to #121 (enforcement / behavior change, not accuracy):** P1 items 5/6/7 (validateLogic backstops for `servers[*].url`, per-server `sync.*` ranges, `sync.baseRetryDelayMs` upper bound) — the schema already expresses these, so the flip enforces them rather than duplicating logic. Also: authoritative cron validation (rejecting `*/0` etc.), the shared-dataDir / duplicate-budget `WARN`->`THROW` promotions, and applying `additionalProperties:false` repo-wide. Unraid: the compose read-only single-file config mount that can defeat first-run bootstrap seeding is a deployment-doc item (not changed here to avoid altering mount semantics); the published Community Applications listing is auto-derived from this in-repo template, so xml parity = listing parity.

## Prioritized fix list (all verified; 0 refuted)


### P1 (9)

- **notifications.ntfy.url** _( schema )_  `CONFIRMED`
  - issue: url is required UNCONDITIONALLY inside the ntfy object (no if-enabled gate), so a disabled stub 'notifications.ntfy:{enabled:false}' fails validation. Inconsistent with telegram/email gating. At hard-fail this would wrongly reject valid configs.
  - fix: Move url out of unconditional 'required' and into an if/then keyed on enabled.const:true (mirror telegram/email). Add minLength>=1 (or format) to url inside that 'then' so an empty/garbage url is rejected only when ntfy is actually enabled.
- **servers[*].sync.autoRetry.*** _( schema+validateLogic )_  `CONFIRMED`
  - issue: Per-server sync.autoRetry.{enabled,maxAttempts,delayMinutes} is read by getSyncConfig (syncService.js:241-243) but has NO schema definition under servers[*].sync. After setting servers[*].sync additionalProperties:false this becomes a hard reject of a legitimate, code-supported override.
  - fix: Add autoRetry (same shape as global sync.autoRetry) to the servers[*].sync schema BEFORE tightening additionalProperties. Until then do not set servers[*].sync additionalProperties:false.
- **notifications.webhooks.telegram** _( schema+example )_  `CONFIRMED`
  - issue: Read by code (notificationService.js:76) as a real send path but absent from the webhooks schema, AND shipped in config.example.json as webhooks.telegram=[]. It only validates because webhooks is open. Tightening webhooks to additionalProperties:false (desired) would hard-reject the shipped example.
  - fix: Decide intent: either (a) add a 'telegram' property to the webhooks schema to legitimize the code path, or (b) treat it as dead and remove the webhooks.telegram path + remove the [] from the example. Resolve BEFORE setting webhooks additionalProperties:false.
- **healthCheck.dashboard.auth** _( schema+validateLogic )_  `CONFIRMED`
  - issue: Descriptions say username/password required when type=basic and token required when type=token, but NO if/then enforces it and no validateLogic backstop. type=basic with blank creds / type=token with blank token passes both surfaces (auth-bypass-shaped). Hard-fail flip would still wrongly ACCEPT a bad/insecure config.
  - fix: Add schema if/then: type=basic -> required username,password (minLength>=1); type=token -> required token (minLength>=1). Also add a validateLogic backstop (schema is advisory now).
- **sync.baseRetryDelayMs** _( validateLogic )_  `CONFIRMED`
  - issue: validateLogic floors at 1000 but has NO upper bound, while schema caps max=10000. Since schema is advisory, the only hard layer (validateLogic) accepts 999999. Drift between the two enforcing surfaces.
  - fix: Add the max=10000 (or chosen ceiling) check to validateLogic to match schema, so the hard-fail layer and schema agree.
- **servers[*].sync.{maxRetries,baseRetryDelayMs,schedule}** _( validateLogic )_  `CONFIRMED`
  - issue: validateLogic only validates GLOBAL config.sync.*, never per-server overrides, yet getSyncConfig merges per-server values that drive real behavior. A per-server maxRetries:999 or garbage schedule passes the only hard layer.
  - fix: Extend validateLogic to iterate servers[*].sync and apply the same range/cron checks it applies to global sync.*.
- **sync.schedule / servers[*].sync.schedule** _( schema )_  `CONFIRMED`
  - issue: Cron pattern is structure-only and accepts out-of-range/zero-step ('99 99 99 99 99','*/0 * * * *') while validateLogic only counts 5 fields. At hard-fail this wrongly ACCEPTS invalid cron that fails at runtime; it also rejects legit named-month/day, '?', 'L' syntaxes.
  - fix: Either parse with the actual scheduler library in validateLogic (authoritative accept/reject) or widen the pattern to allow valid named/extended tokens AND add range validation. Prefer library-parse in validateLogic.
- **servers[*].url** _( validateLogic )_  `CONFIRMED`
  - issue: url format/pattern lives only in the advisory schema; validateLogic only checks falsy. 'foo' or 'ftp://x' passes the hard layer. After flip, behavior depends entirely on whether schema becomes blocking; today there is no real format gate.
  - fix: Add a URL/protocol check to validateLogic (must start http(s)://, non-empty host) so it hard-fails regardless of schema-advisory status.
- **notifications.webhooks.generic[*].url** _( schema )_  `CONFIRMED`
  - issue: required:[url] but no pattern/format/minLength -> empty string '' satisfies 'required'. Hard-fail would wrongly accept a blank webhook URL.
  - fix: Add minLength>=1 and a format:uri (or http(s) pattern) to generic url.

### P2 (10)

- **(schema validation semantics)** _( docs )_  `CONFIRMED`
  - issue: CONFIG.md Validation section + 'Schema Validation Failed' troubleshooting describe schema validation as HARD-FAILING, but post-#115 it is ADVISORY (console.warn, never throws). Documented semantics are stale and will confuse users about why a bad config 'passes'.
  - fix: Update CONFIG.md to state schema validation is currently advisory (warn-not-throw) during the grace period and will become a startup error in a future release; clarify validateLogic checks are the current hard failures.
- **healthCheck** _( docs )_  `PLAUSIBLE`
  - issue: healthCheck/.port/.host/.dashboard.* are in code+schema+example but ABSENT from docs/CONFIG.md (the 'complete reference'); dashboard auth entirely undocumented in README and CONFIG.md.
  - fix: Add a healthCheck section to CONFIG.md covering port, host (with the safe-host/EADDRNOTAVAIL note), and dashboard.auth (none/basic/token + required credentials).
- **notifications.branding / telegram.chatIds / telegram.pollInterval / ntfy.icon** _( docs )_  `CONFIRMED`
  - issue: In schema (and some in example) but undocumented in README and CONFIG.md: branding, telegram.chatIds, telegram.pollInterval, ntfy.icon opt-out.
  - fix: Document these in CONFIG.md/NOTIFICATIONS.md: branding boolean (air-gap behavior), chatIds array alternative to chatId, pollInterval, and ntfy.icon empty-string opt-out.
- **sync.schedule (default)** _( docs )_  `PLAUSIBLE`
  - issue: Documented default is '03 03 */2 * *' but nearly all README/CONFIG examples use '0 2 * * *'/'0 3 * * *', which misleads on the actual default.
  - fix: Align examples to the real default or add an explicit note that examples differ from the default value.
- **logging.redact (always-redact list)** _( docs )_  `CONFIRMED`
  - issue: CONFIG.md prose lists 'credential' (not in schema) and omits encryptionPassword/botToken which the schema description lists. The two redaction lists disagree.
  - fix: Reconcile the documented always-redact list with the schema/logger DEFAULT_REDACT_INDICATORS (single source of truth).
- **logging.redact (not forwarded)** _( code+docs )_  `CONFIRMED`
  - issue: logger.js honors options.redact but syncService.js loggerConfig assembly (95-100) never copies a 'redact' field, so the documented config.logging.redact is unreachable via config.json.
  - fix: Add 'if (config.logging?.redact !== undefined) loggerConfig.redact = config.logging.redact;' to the loggerConfig assembly (and verify per-service loggers if needed).
- **logging.rotation/syslog/performance (per-service)** _( code )_  `CONFIRMED`
  - issue: These are forwarded only to the top-level logger; per-service loggerConfig objects copy only level/format/logDir, so syslog/rotation/performance do not apply to component loggers.
  - fix: Either forward rotation/syslog/performance into the per-service loggerConfig objects, or document explicitly that these are global-logger-only.
- **env:BUDGET_ENCRYPTION_PASSWORD** _( docs )_  `CONFIRMED`
  - issue: CONFIG.md E2EE Docker example shows ${VAR} substitution while itself noting substitution is 'not built-in', i.e. a non-functional/misleading example.
  - fix: Remove the ${VAR} substitution example or replace with the supported approach (literal value or per-server encryptionPassword).
- **env:TELEGRAM_BOT_TOKEN / TELEGRAM_CHAT_ID** _( docs+unraid )_  `CONFIRMED`
  - issue: telegramBot.js env fallbacks are undocumented in CONFIG.md/README and absent from the Unraid template.
  - fix: Document the TELEGRAM_BOT_TOKEN/TELEGRAM_CHAT_ID env fallbacks (and consider exposing in the Unraid xml for the env single-server path).
- **servers[*].sync.baseRetryDelayMs / logging.format / telegram.chatIds / telegram.pollInterval** _( example )_  `PLAUSIBLE`
  - issue: Schema-supported fields never illustrated in config.example.json (per-server baseRetryDelayMs, per-server logging.format, telegram chatIds/pollInterval).
  - fix: Add commented/representative entries to the example so all supported optional fields are discoverable.

### P3 (7)

- **URL patterns (servers/slack/discord)** _( schema )_  `PLAUSIBLE`
  - issue: Patterns are start-anchored but END-unanchored, allowing trailing whitespace/newline/garbage after a valid prefix; servers url allows 'http://' with empty host.
  - fix: Anchor patterns at end ($) and require a non-empty host/path segment.
- **logging.rotation.maxSize / interval** _( schema )_  `PLAUSIBLE`
  - issue: Free-form strings with no pattern: '10MB','1day' validate but misbehave at runtime.
  - fix: Add patterns (e.g. ^[0-9]+[KMG]?$ for maxSize, ^[0-9]+[smhd]$ for interval) matching what the logger actually parses.
- **missing maxima / cross-field** _( schema )_  `CONFIRMED`
  - issue: No max on logging.rotation.maxFiles, performance.thresholds.slow/verySlow, telegram.pollInterval; no cross-field rule verySlow>slow.
  - fix: Add sane maxima and an if/then or validateLogic check that verySlow>slow.
- **empty-string holes (host/dbPath/logDir/tags/redact items/email.to items)** _( schema )_  `PLAUSIBLE`
  - issue: Multiple meaningful strings/array-items accept '' (no minLength).
  - fix: Add minLength>=1 to syncHistory.dbPath, logging.syslog.host, healthCheck.host, notifications.email.host, and item minLength to redact[]/ntfy.tags[]/email.to[]/telegram.chatIds[].
- **servers[*].dataDir (shared dir)** _( validateLogic )_  `PLAUSIBLE`
  - issue: Shared-dataDir across servers is WARN-only despite causing active budget-cache corruption; #116 also targets duplicate-budget for promotion.
  - fix: Promote shared-dataDir (and duplicate-budget per #116 comment at L339) from WARN to THROW in validateLogic.
- **encryptionPassword (E2EE)** _( validateLogic )_  `CONFIRMED`
  - issue: No config-time cross-field check; an E2EE budget missing encryptionPassword only fails at sync time.
  - fix: Optional: add an advisory check or document that E2EE failures surface at sync time (cannot be known from config alone).
- **env:ACTUAL_SYNC_SERVER_DATA_DIR / compose config seeding** _( unraid )_  `PLAUSIBLE`
  - issue: DATA_DIR env is code-supported but absent from Unraid xml/compose; compose bind-mounts config.json read-only single-file, which breaks configBootstrap first-run seeding into an empty dir.
  - fix: Add DATA_DIR to the xml (optional/advanced) and document that compose users must pre-create config.json (or mount the config dir rw) since bootstrap seeding needs a writable dir.
